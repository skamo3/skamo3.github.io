## Skills

- skills 도 적용해서 태스크를 돌릴때마다 skill 을 실행해 정확도를 높여보는게 좋겠다고 판단하여 AGENTS.md를 업데이트함

```markdown
# AGENTS.md

## 목적
- 작업용 최소 규칙만 둔다.
- 문서는 짧게 유지하고, 중복은 합친다.

## Skill 연결
- UE5 에디터 작업, `uasset` / `umap` / 블루프린트 / 머티리얼 작업은 `ue5-editor-control`을 우선 사용한다.
- UE 전반 설계, 개념 정리, 구조 파악, 패턴, 프레임 구조 설명은 `unreal-engine`을 사용한다.
- C++ 코드 변경, 정리, 리팩터링은 `unreal-engine-cpp-pro`를 사용한다.
- GitHub 관련 조사, PR, 리뷰, CI는 `github:github`을 사용한다.
- 로컬 변경 적용, 커밋, 푸시, 드래프트 PR 정리는 `github:yeet`을 함께 고려한다.
- 이미지 생성 또는 레퍼런스 이미지 생성은 `imagegen`을 사용하고 결과물은 `docs/image`에 둔다.

## 코딩 스타일
- `docs/프로그래밍/코딩스타일_UE_요약.md`를 먼저 따른다.
- 기존 코드의 이름, 주석, 구조를 최대한 맞춘다.
- 새 함수나 클래스는 꼭 필요할 때만 만든다.
- 한 함수 안에서 끝낼 수 있으면 쪼개지 않는다.
- 분리는 재사용, 가독성, 중첩 감소에 이득이 있을 때만 한다.
- 지시받은 범위 외 코드는 만들지 않는다.
- 불필요한 `if` 검사는 피하고, 꼭 필요한 최소 조건만 둔다.

## 작업 방식
- 먼저 관련 파일만 읽는다.
- UE 관련 `uasset` / `umap` / 블루프린트 / 머티리얼 분석이나 수정 요청이면, 가능한 경우 먼저 MCP로 자산 구조와 참조를 확인한다.
- 문서 연결은 다음 기준을 따른다.
  - 프로그래밍 규칙과 코딩 기준은 `docs/프로그래밍/코딩스타일_UE_요약.md`
  - 구현 기록과 진행 메모는 `docs/프로그래밍/개발_일지.md`
  - 게임 시스템/구조/진행 규칙은 `docs/기획/게임_기획서.md`
  - 세계관/스토리는 `docs/기획/스토리.md`
  - 생성 이미지 보관은 `docs/image`
- 변경 전 범위와 영향 파일을 짧게 적는다.
- 애매하면 멈추고 사용자에게 확인한다.
- 파일 내용은 필요한 부분만 읽고, 전체 덤프는 피한다.

## 토큰 절감
- 짧게 쓴다.
- 반복 설명은 하지 않는다.
- 출력은 결론부터 쓴다.
- 같은 정보를 두 번 적지 않는다.
- 깊은 함수 체인보다 직접 흐름을 우선한다.
- 불필요한 래퍼 함수, 중간 클래스, 우회 계층을 만들지 않는다.
- 코드와 설명 모두 간결하게 유지한다.

## 검증
- 변경에 필요한 최소 검증만 한다.
- 코드면 관련 빌드/테스트, 문서면 링크와 참조만 확인한다.

## 기록
- `docs/프로그래밍/개발_일지.md`는 작업이 끝났을 때만 갱신한다.
- 개발일지는 `TodoList`와 짧은 누적 기록만 둔다.
- Git 커밋 메시지로 상세 추적하고, 일지는 요약만 적는다.
```

- 이 전에 토큰 사용 및 답변의 정확도를 높이기 위해서 CAVEMAN 프로젝트를 분석 및 일부 적용하기도 했었다.
  - 실제로 효과가 있었음. 답변의 길이가 짧아지고 처리가 훨씬 빨라짐.

- 위 두 내용 적용 후 → 할루시네이션이 많이 사라짐
- 필요없는 파일 수정해서 토큰 날려먹는 일이 적어짐
- 명령 내릴때 해당하는 필요한 Skills를 찾아서 작업하니 잘못된 작업을 하지 않고 컨텍스트 유지에 더 효과적
- 기존에 docs가 많아지면서 codex가 더 많은 파일을 읽기 위해 더 많은 토큰을 사용할 것이라고 생각해 docs 하위를 정리했었다.
- docs를 쓰는게 오히려 토큰 효율적인 면에서 좋다는 조언을 받고 세팅해 봄 → 더 지켜봐야겠지만 아무것도 없을 때보다 사용이 더 쾌적해진건 확실하다.

결과는 더 써보면서 최적화가 필요하겠지만 Skills의 존재를 알고 하네스 엔지니어링과 결합하여 사용해보니 확실히 효율이 높아진 거 같다. -> 기분탓일 수도 있음

unreal skills 중 uasset을 읽을 수 있게해주는 툴도 있는데 아직 직접 명령을 내려보진 못했다. 이것도 사용해보고 어디까지 조작 가능한지 테스트 해봐야겟다. UE 5.8이 나온 시점에서 불필요한 행동일 수도 있지만 용량 문제 및 프로젝트 버전 업 전에 테스트해보고 추후에 버전 올라갈 때 바로 적용할 수 있도록 하는 사전 준비 단계라고 생각하고 작업하고 있다.
