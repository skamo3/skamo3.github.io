이력서 포트폴리오를 다시 정리하면서, 웹페이지로도 만들어보기로 했다. 형태는 Github IO로 구현하기로 정하고, 구현 자체는 Claude Code를 이용해서 진행했다. 레퍼런스 페이지로는 [Badarang](https://badarang.dev/)을 참고했다. 클로드가 어디까지 할 수 있을지 실험해본 건데, 진짜 잘한다.

색은 내가 초록 계열을 좋아해서 색 코드부터 맞췄다. 기본 배경은 화이트모드로 주었는데, 다크모드는 하이라이트에는 좋지만 글이 많으면 오히려 산만해 보일 수 있기 때문이다. 포트폴리오는 특성상 글이 많다. 그래서 하얀 배경이 더 나은 것 같다.

## 기본 구성

시작할 때부터 하네스 엔지니어링을 적용해서 시작했다. `AGENTS.md`를 기본으로 잡고, `docs` 내부에 필요한 가이드와 작업 시 생기는 기록들을 적도록 잡았다. 기존에 Zolta 프로젝트에 적용했던 하네스 엔지니어링 기법을 참고해서, 웹 블로그 형태에 맞도록 서칭하게 했다.

Claude를 위한 세팅에만 일간 토큰을 40퍼센트 넘게 사용했다.. ㄷㄷ 근데 가치있는 행동이었다. 4일차쯤 지난 현재에도 클로드가 컨텍스트 파악을 제대로 하면서 휘청거리는 일 없이 작업을 제대로 한다.

## 계획 모드 사용하기

클로드의 계획 모드는 전부터 많이 들어봤다. 토큰을 아끼기 위해선 이 방법이 가장 효과적이라고 하여 바로 사용해봤다.

특정 명령을 넣으면 어떻게 바꿀 것인지 쭉 나열해주고, 드래그해서 댓글을 남겨 특정 부분 수정도 가능하다. 사실 CSS, JS 기반 코드라 다 이해하긴 어렵다 ㅋㅋ 그냥 내용만 보고 이 정도쯤 나오겠구나 하고, 내용에서 틀린 부분이 있으면 댓글 혹은 수정 버튼을 눌러서 전체적인 계획을 손본다.

## 포폴 내용 추가하기

기존의 포폴들은 내용이 애매해서, 나만의 기준을 잡고 이를 토대로 작성하고자 했다. 무엇을 했는가, 어떻게 했는가, 왜 했는가를 기준으로 잡았고, 이력서에 들어갈 문구는 추가로 문제 + 해결 + 결과 + 도메인을 한 줄로 표현할 수 있는 방법을 찾도록 했다.

블로그의 형태로 추가될 것이기 때문에, 줄글로 표현하면서도 내용이 한눈에 들어오게 쓰는 것을 핵심으로 생각했다. 그러면서도 내용 자체에 빠진 부분은 없도록 하는 것이 핵심이었다.

내가 프로젝트를 진행하면서 어떤 생각과 어떤 흐름으로 그 결론에 도달했는지 써내는 것을 중점으로 생각하고, 클로드에게 빠진 부분이 있다면 나에게 반문하도록 했다. 오래된 프로젝트는 기록하는 법을 모를 때여서 다시 기억해내는 게 어려운 때도 있었다 ㅜㅜ

작업 중 애매한 부분이 생기면 md 파일에서 직접 수정해서, 자잘한 변경에는 토큰이 안 쓰일 수 있도록 했다.

## 블로그 탭 제거 및 사이트 추가

기존 티스토리 블로그 내용을 그대로 긁어오도록 했다. 다만 기존 티스토리에선 되어있지 않던 카테고리별 분류 버튼을 추가했다.

메인 페이지에서 보여질 블로그 글도 중요할 것 같았다. 최신 글만 띄워놓기보다는 잘 쓴 글을 띄워놓는 게 더 좋을 것 같은데, 고민 끝에 메인 페이지에서 블로그 섹션 자체를 삭제하고 상단 배너에만 링크로 띄워놓는 방식으로 바꿨다. 포트폴리오 메인 페이지에서 굳이 정보가 많을 필요도 없다는 판단이었다.

## 회사별로 다르게 보여주기 — 하이라이트 링크

포트폴리오 사이트를 여러 회사에 지원할 때 링크로 공유하는데, 회사마다 관심 있는 기술 스택/프로젝트가 다르다. 그런데 프로젝트 목록은 고정된 순서로만 정렬돼 있어서, 검토자가 어떤 프로젝트를 먼저 봐야 할지 알기 어려울 수 있겠다 싶었다.

그래서 `?highlight=슬러그1,슬러그2` 형태의 URL 쿼리를 만들면, 해당 프로젝트 카드에 별 배지가 붙고 그리드 맨 앞으로 재배치되도록 했다. 링크만 만들면 되고 별도 파일 관리가 필요 없는 방식이라 회사마다 다른 링크를 그때그때 만들어 보내면 된다.

만들면서 버그를 두 개 만났다. 하나는 정렬 로직 — 여러 프로젝트를 하이라이트하면 쿼리에 먼저 적은 게 맨 앞에 와야 하는데, 실제로는 마지막에 적은 게 맨 앞으로 오는 순서 반전 버그가 있었다. 카드를 `insertBefore`로 하나씩 맨 앞으로 옮기는 로직이 처리 순서상 뒤집혀서 생긴 문제였다.

다른 하나는 별 배지 이미지의 클릭 영역 문제였다. 배지를 카드 모서리에 크게 걸치도록 디자인했는데, 이미지의 투명한 여백까지 전부 클릭 히트박스로 잡혀서 옆 카드를 클릭하려다 엉뚱하게 하이라이트된 카드가 클릭되는 문제가 있었다. `pointer-events: none`을 배지에 걸어서, 장식용 이미지가 클릭을 아예 가로채지 않도록 고쳤다. 예쁘게 만드는 것보다 실제 클릭 영역이 보이는 대로 맞는지가 더 중요했다.

## 히어로 자기소개 다시 쓰기

메인 페이지 맨 위 자기소개 문구가 계속 마음에 걸렸다. 이력서 Summary를 그대로 옮기면, 이력서를 보고 들어온 사람 입장에서는 "이미 본 내용"처럼 느껴질 것 같았다.

그래서 먼저 자소서 두 편을 다시 검토했다. 반복되는 단어를 줄이고, "신입임에도"처럼 스스로를 낮추는 표현을 걷어내고, 전체적으로 더 자신감 있는 톤으로 다듬었다. 그 소재를 바탕으로 히어로 문구를 프로즈(성향을 보여주는 짧은 문장 4줄) + 불릿(실제 성과 3개) 구조로 여러 버전 실험했다. 사진도 정사각형 크롭에서 세로형으로 바꾸고, 크기·위치·밝기까지 몇 번을 더 조정했는지 모른다.

## 문의하기 폼 만들기 — 정적 사이트의 한계

Github Pages는 정적 사이트라 서버가 없다. 그래서 폼에 이름/이메일/내용을 입력해도, 그걸 자동으로 이메일로 보내주는 기능을 붙이기가 쉽지 않았다.

처음엔 `mailto` 방식으로 시도했다 — 버튼을 누르면 내 기본 메일 앱이 열리고, 거기서 다시 "보내기"를 눌러야 실제로 전송되는 방식이다. 완전 자동은 아니지만 서버 없이 가능한 가장 안전한 방법이었다.

이후 Web3Forms라는 서비스를 찾았다. 비밀번호도, 대시보드 로그인도 없이 이메일만 입력하면 즉시 Access Key를 받을 수 있고, 이 키는 비밀 API 키가 아니라 이메일의 별칭 같은 값이라 공개돼도 안전하다고 문서에 명시돼 있었다. 이 키를 폼에 넣고 API로 직접 POST하도록 바꾸니, 버튼 한 번 누르면 조용히 이메일로 도착하는 진짜 폼이 완성됐다.

## 모바일 레이아웃 다시 잡기

한참 데스크톱 화면만 보고 작업하다가, 모바일 레이아웃을 한 번에 정리할 때가 됐다 싶어 손을 봤다.

가장 먼저 눈에 띈 건 네비게이션이었다. "소개/프로젝트/블로그/문의하기" 4개 메뉴와 테마 토글이 좁은 화면에서는 한 줄에 다 들어가지 못하는 문제가 있어서, 640px 이하에서는 햄버거 버튼으로 접히도록 바꿨다. 하이라이트 별 배지도 데스크톱 크기 그대로면 모바일에서 화면 밖으로 잘려나가서, 화면 크기에 맞춰 따로 축소했다.

이 작업을 하면서 클로드가 완벽하지 않다는 것도 느꼈다. 세션 내내 브라우저 미리보기 도구 연결이 잘 안 돼서, 실제 화면을 직접 보여주지 못하고 코드만 보고 추론해서 수정한 다음, 내가 직접 확인해서 피드백을 주는 식으로 반복해야 했다. 그래도 뭐가 안 되는지 솔직하게 말해줘서, 어디를 내가 직접 봐야 하는지는 명확했다.

## 카피 톤에 위트 넣기

"문의하기"라는 말이 너무 밋밋하게 느껴져서, 클로드랑 같이 여러 표현을 브레인스토밍해봤다. 접선 시도, 핑 보내기, 커밋 요청, 매칭 신청 같은 후보들을 놓고 고민하다가, 결국 게임 개발자라는 정체성과 맞닿는 "매칭신청"으로 상단 메뉴와 버튼을 바꾸고, 실제 폼 전송 버튼은 "Ping"으로 남겨서 네트워킹 은유 하나로 이어지게 정리했다. 다만 실제 액션이 일어나는 버튼(전송)은 헷갈리면 안 되니 위트보다 명확성을 우선했다.

## Git 관리

Github IO이므로 당연히 github에 올려야 하는데, 이것도 클로드를 이용해 진행했다. 좀 나중에 체크해서 몰랐는데, 커밋 메시지에 본인 이름과 함께 Claude 이름도 같이 껴 들어가고 있었다 ㅋㅋㅋㅋ 커밋 메시지는 항목별로 잘 구분해서 넣어줘서, 추후 디버깅에도 확실히 요긴하게 쓰일 듯하다.

## 이 글도 클로드가 썼다

사실 이 글도 클로드한테 시켰다. 지금까지 작업한 내용과 내가 작업하면서 메모해둔 것들을 PDF로 뽑아서 클로드에 입력하고, 그 내용을 블로그 형태로 정리하게 했다.

진짜 잘 쓴 블로그처럼 정리된 게 신기했다. 아직 AI가 썼다는 느낌이 살짝 묻어있긴 하지만, 간단하게 메모해둔 것들로 이 정도 글을 만들어준다는 게 시간 절약의 핵심 아닐까 싶다. 블로그는 회고록과 비슷하니까, 이걸 다시 읽어보면서 내가 뭘 했는지 되새길 수 있는 콘텐츠가 됐다는 점에서 AI를 유용하게 사용한 것 같은 기분이 들었다..!

## 마치며

이 글을 어느 카테고리에 넣을지도 잠깐 고민이었다. 다중 태그로 만들면 더 정확하게 분류할 수 있었겠지만, 그러면 애초에 카테고리를 나눠서 보려던 목적 자체가 흐려질 것 같아서, 그냥 카테고리를 하나(`AI`) 새로 추가하는 쪽을 택했다. 완벽한 구조보다 실제로 쓰기 편한 구조를 택한 셈이다.

정석적인 방법은 아니었을지 몰라도, Claude Code와 계속 대화하면서 픽셀 단위, 문구 단위까지 반복해서 다듬다 보니 결과물이 꽤 마음에 든다.
