# 기술 스택 검토: Jekyll vs React/Next.js/Astro

"Jekyll도 결국 React/JS로 가는 거 아니냐"는 질문에 대한 조사/결론 기록.

## 오해 정정
Jekyll은 **빌드 시점에 Ruby로 동작**해서 순수 HTML/CSS(+필요한 곳만 최소 JS)를 만들어내는 정적 사이트 생성기다. 배포되는 결과물에는 React나 Next.js 런타임이 전혀 없다 — 브라우저는 그냥 평범한 HTML을 받는다. "결국 JS로 간다"는 전제는 틀렸다. React 계열로 갈아탄다고 해서 더 발전된 단계로 가는 게 아니라, **다른 도구를 쓰는 선택지**일 뿐이다.

## React/Next.js로 직접 만들 경우
가능하다. Next.js는 `next.config.js`에 `output: "export"`를 설정하면 완전한 정적 HTML/CSS/JS로 빌드되고(`out/` 폴더), 이걸 GitHub Pages에 그대로 올릴 수 있다.
- User Page(`skamo3.github.io`, 루트 경로)라 `basePath`/`assetPrefix` 설정 불필요(프로젝트 페이지였다면 필요)
- `getServerSideProps`, API Routes 등 서버 기능은 정적 export에서 사용 불가
- `public/.nojekyll` 파일 필요 (GitHub가 `_`로 시작하는 폴더를 Jekyll 규칙으로 무시하는 걸 방지)
- 배포는 지금 만든 것과 거의 동일한 GitHub Actions 워크플로우로 가능 (`out/` 폴더를 `upload-pages-artifact`로 업로드)

## Astro (참고용 대안)
정적 콘텐츠 중심 사이트(포트폴리오/블로그)에 최적화된 프레임워크. 기본적으로 **JS를 0바이트** 배포하고(필요한 인터랙션 부분만 선택적으로 "island"로 hydration), React 컴포넌트도 필요하면 부분적으로 섞어 쓸 수 있다. GitHub Pages 배포 설정도 매우 간단.

## 비교 요약
| | Jekyll (현재) | Next.js (static export) | Astro |
|---|---|---|---|
| 초기 JS 용량 | 0 | ~87KB (React 런타임) | 0 (필요한 곳만) |
| 빌드 속도 | 빠름 | 보통 | 매우 빠름 |
| 컴포넌트 재사용 | Liquid include | React 컴포넌트 | 어떤 프레임워크든 혼합 가능 |
| GitHub Pages 배포 | 이미 완료 | Actions 재구성 필요 | Actions 재구성 필요 |
| 학습/전환 비용 | 없음(이미 구축됨) | 중간 (Next 개념 학습) | 낮음~중간 |
| 이 프로젝트와의 궁합 | 콘텐츠 중심 포트폴리오+블로그에 충분 | 오버스펙 (동적 기능 불필요) | 콘텐츠 중심이라 이론상 적합하지만 재구축 비용 발생 |

## 결론 / 추천
김주찬님은 **게임 클라이언트(엔진/그래픽스) 개발자**이고, 이 사이트의 역할은 프로젝트를 잘 보여주는 것이지 웹 프레임워크 실력을 증명하는 게 아니다. React/Next.js 사용 여부가 채용 평가에 미치는 영향은 거의 없고, 오히려 정적 콘텐츠 사이트에 무거운 JS 프레임워크를 얹는 건 로딩 속도만 손해다.

**현재 Jekyll 구축 상태를 유지하는 것을 추천.** 이미 커스텀 디자인으로 스캐폴딩 완료, 로컬 테스트 환경 구축 완료, 실제 배포까지 검증 완료된 상태라 갈아탈 실익이 없다.

다만 향후 다음과 같은 요구가 생기면 그때 Astro/Next 전환을 재검토할 만하다:
- 프로젝트 카드에 필터링/검색 등 복잡한 클라이언트 인터랙션이 필요해질 때
- React 컴포넌트 재사용이 많은 다른 프로젝트와 코드/디자인 시스템을 공유하고 싶을 때
