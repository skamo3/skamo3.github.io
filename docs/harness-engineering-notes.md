## 핵심 개념
- **Agent = Model + Harness**. 모델이 아닌 모든 것(규칙, 툴, 스킬, 메모리, 피드백 루프)이 하네스.
- Hashimoto(하네스 엔지니어링 개념 주요 제안자)의 원칙: "에이전트가 실수할 때마다, 그 실수가 다시 일어나지 않도록 만드는 것"이 하네스 엔지니어링.
- "좋은 모델 + 평범한 하네스" 보다 "평범한 모델 + 좋은 하네스"가 더 나은 결과를 낸다.
- 하네스 산출물(`CLAUDE.md`/`AGENTS.md`, 스킬 파일, 훅 스크립트)은 레포에 커밋되는 파일이므로, 팀원이나 모델이 바뀌어도 축적된 하네스는 유지된다.

## 4가지(또는 5가지) 구성요소 (목차 기준 재구성)
1. **지시문서(Instruction Docs)** — `AGENTS.md`/`CLAUDE.md`: 프로젝트 설명, 컨벤션, 빌드 명령
2. **아키텍처 제약(Architecture Constraints)** — 디렉토리 구조, 허용된 패턴/금지된 패턴
3. **피드백 루프(Feedback Loop)** — 실수 기록, 테스트/빌드 검증, 훅(hooks)
4. **지식 저장소(Knowledge Store)** — `docs/` 폴더, 메모리 시스템
5. **가비지 컬렉션(Garbage Collection)** — 오래되거나 틀린 지식/문서를 주기적으로 정리

## 이 프로젝트에 적용한 방식
- `AGENTS.md`(루트) = 지시문서. Claude Code는 `CLAUDE.md` 외 `AGENTS.md`도 인식하므로 Codex와 동일한 컨벤션 사용 가능.
- `docs/profile-source.md` = 지식 저장소 (콘텐츠 소스 오브 트루스)
- `docs/site-architecture.md` = 아키텍처 제약/결정 기록
- `AGENTS.md`의 "피드백 루프(실수 기록)" 섹션 = 피드백 루프. 앞으로 작업 중 반복되는 실수가 발견되면 이곳에 계속 추가.
- 가비지 컬렉션은 아직 적용 대상 없음(프로젝트 초기) — 문서가 실제 코드/구조와 어긋나기 시작하면 주기적으로 정리 필요.
