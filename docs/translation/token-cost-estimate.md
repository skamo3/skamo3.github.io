# 번역 파이프라인 도입 검토: 토큰 비용 예측 (적용 전 조사, 미적용 상태)

**상태: 조사만 완료, 적용 안 함.** 사용자가 이 예측을 보고 도입 여부를 판단할 예정.

## 제안된 방식
한국어 입력 → 영어로 번역 → (영어로 사고/답변) → 답변을 다시 한국어로 번역해서 전달. 영어가 한국어보다 모델 정확도가 높다는 전제.

## 구현 방식에 따라 비용 차이가 크다

### 방식 A — 한 번의 응답 안에서 내부적으로 처리 (권장)
Claude가 한 번의 생성(=한 번의 API 호출) 안에서 "한국어 입력을 영어로 이해 → 영어로 답변 초안 → 한국어로 최종 변환"까지 끝내고 한국어만 출력.
- **입력 토큰**: 거의 증가 없음 (대화 컨텍스트를 한 번만 전송)
- **출력 토큰**: 오늘날의 "한국어 답변만" 생성 대비, "영어 답변 + 한국어 재번역" 두 벌을 생성해야 함
  - Claude의 토크나이저 특성상 한국어는 동일한 내용이라도 영어보다 토큰을 더 많이 씀 (경험적으로 약 1.4~2배)
  - 즉 영어 답변 자체는 한국어 답변보다 짧은 토큰 수(대략 0.6~0.7배)로 끝나지만, 그 위에 한국어 재번역(원래 한국어 답변과 비슷한 토큰 수)이 추가로 붙음
  - **예상 증가폭: 턴당 출력 토큰 약 +50~90%**
- 입력이 큰 턴(파일을 많이 읽거나 긴 대화 맥락)에서는 출력 비중이 작아서 전체 턴 비용 증가폭은 상대적으로 작음(예: 입력 10k + 출력 1k 턴이라면 전체는 약 +5%대)
- 출력이 큰 턴(설명/문서 작성이 많은 턴, 지금 이 세션 같은 경우)에서는 체감 증가폭이 더 크게 남

### 방식 B — 번역/답변/재번역을 별도 메시지(호출)로 분리 (비권장)
매 단계를 별도의 API 호출로 나누면, 대화가 길어질수록 **매 호출마다 전체 대화 히스토리를 다시 전송**해야 함.
- 프롬프트 캐싱(5분 TTL)이 잘 맞아떨어지면 반복 전송 비용을 캐시 히트 가격(정가 대비 훨씬 저렴)으로 줄일 수 있지만, 별도 엔지니어링 없이는 보장되지 않음
- 캐싱이 제대로 안 걸리면 대화가 길어진 지금 같은 세션에서는 **입력 토큰이 호출 횟수만큼(3배 안팎) 반복 청구**되어 전체 비용이 **약 2~4배**까지 뛸 수 있음
- 그래서 이 방식은 권장하지 않음

## 종합 예측
- **방식 A로 구현 시**: 턴당 총 토큰 사용량 대략 **+10~90%** (턴의 입출력 비율에 따라 편차 큼, 평균적으로는 +30~50% 선으로 예상)
- **방식 B로 구현 시**: **+100~300%** (권장하지 않음)

## 참고: 실익에 대한 의문
Claude 최신 모델은 한국어 처리 정확도가 이미 상당히 높은 편이라, 일반적인 대화/코딩 작업에서 번역 파이프라인으로 얻는 정확도 개선폭은 크지 않을 가능성이 있다. 반면 법률·전문 용어·미묘한 뉘앙스가 중요한 특수 케이스에서는 체감 효과가 있을 수 있음.

## 도입한다면
- 방식 A로만 구현
- 매 턴이 아니라 정확도가 특히 중요한 질문에만 선택적으로 적용하는 것도 비용 대비 효과적인 절충안
